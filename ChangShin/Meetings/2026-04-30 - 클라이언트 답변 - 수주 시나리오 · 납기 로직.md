# 2026-04-30 · 클라이언트 답변 — 수주 시나리오 · 납기 로직

> 상위 문서: [[00 - Meetings (Index)]]

> [!abstract] TL;DR
> 수주 화면 입력은 **기제품 / 신제품 두 시나리오**로 분기. 기제품은 `자재코드 + 수량`, 신제품은 **가장 유사한 제품의 자재코드**를 입력해서 예상 납기를 받는다. 신제품 흐름엔 **금형 설계+개발 일정**(평균 45일, 변동 큼)이 추가된다. 납기 계산식 자체는 **앞으로 구현해야 할 부분**이며, 부자재·공정별 평균 소요 시간은 **생산관리팀장 추가 인터뷰** 후 확정 예정. **외주업체 생산계획은 직접 받을 수 없고**, 시스템 내 기수주 건 기반으로 지연을 추정하는 구조로 가야 한다. **수주 확정 후 프로세스는 온라인 미팅에서 상세 설명** 받기로.

---

## 1. 핵심 결정 / 답변 (Critical Outcomes)

### A. 수주 시나리오 = 2-way 분기 (기제품 vs 신제품)

| 시나리오 | 입력 항목 | 비고 |
|----------|-----------|------|
| **A. 기제품 수주** | 자재코드 + 수량 | 납품 이력이 있는 제품. **후가공만 변경되는 신제품도 이 흐름** 으로 처리 |
| **B. 신제품 수주** | "가장 유사한 제품"의 자재코드 + 수량 | 납품 이력이 없는 신제품. 금형 설계+개발 일정이 추가됨 |

> [!note] 화면 UX 함의
> 수주 입력 화면 첫 단계는 **시나리오 선택(기제품/신제품)** → 그에 따라 다음 입력 폼이 갈라지는 구조로 설계.
> 신제품도 자재코드 입력을 받지만 그 의미는 "유사 제품 참조"이며, 이를 화면 라벨/도움말로 명확히 구분해야 함.

---

### B. 예상 납기일 로직 = 미정 (구현 필요 + 추가 인터뷰 예정)

- **신제품 금형 개발 평균**: ~45일
  - 변동 요인: 금형 **소재**, **난이도**, **크기**, **업체별 capa (금형 개발)** 가 가장 큰 변수
- **기제품**: 부자재 / 공정별 평균 소요 시간이 **생산관리팀장 머릿속**에 있음
  - 클라이언트(서주현 팀장)가 **추가 인터뷰** 진행 후 변수 정리해서 공유 예정
- **외주업체 생산계획 직접 수신 불가** → 그 대신 **시스템 내 현재 수주 건들을 기반**으로 일정 지연 추정 시스템을 갖춰야 함

> [!warning] 함의
> Module 1 의 핵심 알고리즘은 결국 "기수주 점유율 + 공정별 평균 소요 + 업체 capa" 의 **조합 함수**. 데이터 모델은 이 변수들을 모두 표현 가능하게 설계해야 함.
> - SKU ↔ 공정 ↔ 외주처 매핑 테이블 필요
> - 공정별 평균 소요 시간 마스터 필요 (변수: 소재/난이도/크기/업체)
> - 외주처 capa 마스터 필요 (금형 개발 capa, 일반 생산 capa)
> - 현재 기수주 점유 상태를 실시간 또는 배치로 집계할 수 있는 구조

---

### C. 수주 확정 후 프로세스 = 온라인 미팅에서 설명 예정

- 카톡으로는 정리하지 않고, **온라인 미팅 잡아서 상세 설명** 진행
- 본인이 미팅 일정을 잡아 진행해야 함

## 2. 논의 내용 (원문)

### 본인 질의

```
영업과 수주예측/확정 모듈 사이 '수주 시나리오 입력', '예상 납기일', '수주 확정' 부분이
이 워크플로우의 시작점이라고 인지됩니다.

1. 수주 시나리오라는 건 어떤 데이터를 입력받는지 궁금합니다.
2. 예상 납기일 관련 내용을 살펴보니 계산식 또는 로직의 언급이 있던데,
   이 부분 공유가 가능할까요?

그리고 '수주 확정' 시 진행되는 프로세스에 대해서 전반적으로 설명을 한번 주시면
개발 시 많은 도움이 될 것 같습니다.
```

### 클라이언트 답변 (서주현 팀장 · 모니터 딜로이트)

```
1. 두가지 시나리오가 있습니다.

A) 기제품 수주: 납품 이력이 있는 기제품 수주.
   제품(자재코드) 선택하고 수량 기입.
   (후가공만 변경되는 신제품도 기제품 수주랑 유사합니다)

B) 신제품 수주: 납품 이력이 없는 기제품 수주.
   이때는 금형 설계+개발 일정이 추가되어야 합니다.
   이때는 해당 제품과 가장 유사한 제품 자재코드를 넣고
   예상 납기일을 확인하도록 구현해야할 것 같습니다.

2. 예상 납기일 로직은 앞으로 구현해야하는 부분입니다.
   뭐 예시적으로 신제품 금형 개발의 경우 평균적으로 45일정도 소요되지만
   금형의 소재, 난이도, 크기 그리고 제일 중요한 업체별 capa (금형 개발)
   인하여 변경됩니다.

   기제품 발주라도 부자재/공정별로 소요되는 평균적인 시간이
   생산관리팀장님 머리 속에 있으실거라 생각됩니다.
   해당 부분을 제가 추가로 더 인터뷰 해보겠습니다.
   어떤 변수들로 인하여 해당 부분이 변화하는지에 대해서요.
   + 저희가 외주업체의 생산 계획을 받을 수는 없어도
   최소한 시스템 내에 현재 수주되어있는 건들을 기반으로
   얼마나 일정이 지연될지는 고려하는 시스템이 구축되었으면 합니다.

수주 확정 후 프로세스는 온라인 미팅 잡아주시면 제가 상세히 설명드리겠습니다.
```

## 4. 본인 영향 (Personal Takeaways)

- **Module 1 entity 골격 작성 가능 범위가 명확해짐**: 시나리오 분기 + 입력 필드(자재코드 + 수량)는 확정. 채번 규칙·고객사 마스터·납품 주소 등은 여전히 미확정이지만 **수주 입력 화면 1차 와이어프레임은 이 정보로 시작 가능**.
- **납기 계산식을 코드에 박을 수 없음**: 인터뷰 결과 도착 전엔 `OrderDeliveryEstimate` 같은 결과 저장 엔티티 + 비어있는 계산 서비스 인터페이스만 만드는 수준이 적절. 알고리즘 본체는 Phase 2 로 분리.
- **마스터 데이터 추가 필요**: 공정 마스터, 외주처 마스터(capa 포함), SKU↔공정↔외주처 매핑이 새로 추가될 영역. SKU 마스터([[Infrastructure/31 - Decision Log#D-021]])는 시작점이지만 충분치 않음.
- **외주처 생산계획 미수신 전제** → "시스템 내 기수주 점유율"이 납기 추정의 1차 입력. Order/OrderItem 의 상태·일정 컬럼이 정확하게 잡혀야 다른 모듈도 작동.
