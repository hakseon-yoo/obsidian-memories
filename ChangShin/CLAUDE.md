## 0-2. Claude가 자주 쓰는 검색 패턴

프론트매터가 일관되면 아래 검색이 강력하다. Claude가 문서를 찾을 때 활용:

| 목적 | 검색 예시 |
|------|----------|
| 특정 프로젝트 전체 문서 | `grep "project: AX-2" -r` |
| 특정 모듈 문서 | `grep "module: M1" -r` |
| 미완 상태 문서 | `grep "status: draft" -r` |
| 특정 태그 | `grep -r "^\s*- AX-2$" *.md` (프론트매터 tags 리스트) |
| 특정 날짜 이후 생성 | `grep "created: 2026-0[4-9]" -r` |

새 문서 만들 때 프론트매터 필드를 **생략하지 말 것** — 검색/필터링 가치가 사라진다.

## 1. 사용 환경 · 기본 원칙

- 이 폴더(`ChangShin`)는 옵시디언 Vault의 한 부분이다.
- 작성하는 md는 기본적으로 **옵시디언에서 열람·편집되는 문서**로 본다.
- 다만 예외: 프로젝트 내부용 기술 문서(`README.md`, 코드 문서 등)는 **일반 마크다운 양식**으로 작성해도 된다. 이 경우에도 프론트매터는 붙이는 것을 권장한다.
- 판단 기준: *"이 md 파일을 옵시디언 그래프 뷰/검색에서 연결될 가치가 있는가?"* → Yes면 옵시디언 양식으로 작성.

### 작성 시 사고 순서

1. **위치**: 어느 폴더에 들어가야 하는가? 기존 폴더 구조를 먼저 훑어보고 결정.
2. **파일명**: 옵시디언에서 위키링크로 참조하기 좋은 이름인가? (공백/특수문자 최소화, 번호 prefix 고려)
3. **프론트매터**: 필수 필드 + 이 노트에 의미 있는 메타데이터.
4. **연결**: 다른 노트(`parent`, `related`, 본문 위키링크)와 어떻게 연결되는가?
5. **구조**: 짧은 노트인지, MOC 인덱스형인지, 개별 원자 노트인지.

## 2. 프론트매터 (YAML Properties)

모든 옵시디언 노트 상단에는 YAML 프론트매터를 둔다. 옵시디언 "Properties" 기능이 이 필드들을 자동 파싱한다.

### 필수 필드

```yaml
---
title: 문서 제목
created: 2026-04-23
tags:
  - 태그1
  - 태그2
---
```

- `title`: 파일명과 달라도 된다. 사람이 읽기 좋은 제목.
- `created`: **반드시 ISO 날짜 형식(`YYYY-MM-DD`)**. 파일 시스템 시간이 아닌, "노트 내용 기준" 생성일.
- `tags`: **리스트 형식**으로. 단일 태그라도 `- 태그명` 형식 유지.

### 선택 필드 (맥락 따라 추가)

| 필드 | 용도 | 값 형식 |
|------|------|---------|
| `parent` | 상위 문서 (MOC 등) | `"[[문서명]]"` (따옴표 필수) |
| `project` | 소속 프로젝트 | 문자열 (`AX-2` 등) |
| `version` | 문서 버전 | `v1.0` |
| `source` | 원본/출처 | 파일명, URL, 메모 |
| `audience` | 대상 독자 | `개발팀`, `경영진` 등 |
| `status` | 진행 상태 | `draft` / `review` / `done` |
| `updated` | 마지막 수정일 | `YYYY-MM-DD` |
| `aliases` | 다른 이름으로도 링크 가능 | 리스트 |
| `related` | 관련 문서들 | 리스트 of `"[[...]]"` |
| `module` | 모듈 코드 등 | `M1`, `M2` … |

### 작성 규칙

- `"[[...]]"`처럼 위키링크를 값에 넣을 때는 **반드시 따옴표로 감싼다** (YAML 파싱 오류 방지).
- 한글 키는 쓰지 않는다 (옵시디언 Properties 호환성).
- 불필요한 필드는 넣지 않는다. "있으면 꺼내 쓸 수 있다"가 아니라 **"검색·필터에 진짜 쓰는가"** 기준.

## 3. 위키링크 · 임베드

문서 간 참조는 옵시디언 위키링크를 사용한다. **마크다운 링크(`[text](url)`)는 외부 URL에만 사용**한다.

### 링크 문법

| 용도 | 문법 | 예시 |
|------|------|------|
| 기본 파일 링크 | `[[문서명]]` | `[[02 - Module 1 · 납기일 예측]]` |
| 표시 이름 별칭 | `[[문서명\|보여줄 이름]]` | `[[02 - Module 1 · 납기일 예측\|납기 예측]]` |
| 섹션(헤딩) 링크 | `[[문서명#섹션명]]` | `[[08 - 질문#Module 1 — 납기 예측]]` |
| 블록 참조 링크 | `[[문서명#^블록ID]]` | `[[노트명#^abc123]]` |
| 같은 문서 내 섹션 | `[[#섹션명]]` | `[[#목차]]` |
| 같은 문서 내 블록 | `[[#^블록ID]]` | `[[#^def456]]` |

### 임베드 (inline 삽입)

링크 앞에 `!`를 붙이면 해당 내용이 **미리보기에 직접 렌더링**된다.

- `![[문서명]]` → 문서 전체 임베드
- `![[문서명#섹션]]` → 특정 섹션만 임베드
- `![[문서명#^블록ID]]` → 특정 블록만 임베드
- `![[이미지.png]]` → 이미지 삽입
- `![[이미지.png|400]]` → 이미지 크기 지정 (너비 400px)
- `![[파일.pdf]]` → PDF 미리보기 삽입

### 블록 ID 부여

특정 문단/줄을 다른 곳에서 참조하고 싶으면 해당 줄 끝에 `^블록ID`를 붙인다.

```markdown
이 문장은 다른 노트에서 참조할 수 있다. ^key-insight
```

→ 다른 노트에서 `[[노트명#^key-insight]]`로 링크 가능.

### 링크 작성 원칙

- 존재하지 않는 노트에 링크 거는 것도 허용 (옵시디언은 "unlinked" 상태로 표시하고, 클릭 시 빈 노트를 만든다).
- 긴 파일명이 부담스러우면 `|별칭`으로 짧게 표시.
- 본문에서 자주 언급되는 핵심 용어는 **위키링크로 감싸면 그래프 뷰에서 자연스러운 연결망**이 된다.

## 4. 콜아웃 (Callouts)

정보 유형별로 시각적으로 구분할 때 사용한다. 옵시디언에서 색상·아이콘이 자동 적용된다.

### 주요 유형

```markdown
> [!note] 제목
> 일반 안내

> [!tip] 제목
> 팁 · 추천 · 베스트 프랙티스

> [!info] 제목
> 참고 정보 · 배경 지식

> [!important] 제목
> 중요 · 놓치면 안 되는 내용

> [!warning] 제목
> 주의 · 리스크

> [!danger] 제목
> 심각한 경고 · 치명적 이슈

> [!success] 제목
> 완료 · 성공 · 긍정

> [!failure] 제목
> 실패 · 미달 항목

> [!summary] 제목
> 요약 (abstract/tldr과 동일)

> [!question] 제목
> 확인 필요 질문 · 미해결

> [!example] 제목
> 예시 · 케이스

> [!quote] 제목
> 인용 · 출처 있는 발언

> [!bug] 제목
> 버그 · 알려진 이슈

> [!todo] 제목
> 할 일 · 액션 아이템
```

### 접기(Fold) 옵션

- `> [!note]+ 제목` → 기본 펼침 (접을 수 있음)
- `> [!note]- 제목` → 기본 접힘 (펼칠 수 있음)
- 긴 부가 설명/참고자료에 활용.

### 중첩 콜아웃

```markdown
> [!note] 바깥
> 본문
> > [!warning] 안쪽
> > 중첩된 주의
```

### 사용 가이드

- 본문의 핵심 흐름이 아닌 **주의·요약·보조 정보**에 사용.
- 과용하면 오히려 가독성이 떨어진다. 한 섹션에 콜아웃은 1~2개를 권장.
- 체크리스트형 액션은 `[!todo]` 또는 본문의 체크박스로.

## 5. 태그 시스템

### 원칙

- **프론트매터 `tags:` 리스트를 1순위로 사용**한다.
- 본문 인라인 태그(`#태그명`)는 특정 문단만 태깅해야 할 때만 사용.
- 태그는 **검색·필터·Dataview 쿼리에 쓰는가**를 기준으로 선별.

### 네이밍

- 공백 대신 `-` 또는 붙여쓰기 (`생산관리`, `납기-예측`).
- 대소문자 혼용 피하기 — 모듈 코드는 `Module1`, `AX-2`처럼 일관되게.
- 한글·영어 혼용은 괜찮지만, 같은 개념은 한 가지 표기로 통일.

### 계층 태그

`/`로 구분하면 옵시디언 태그 패널에서 트리 구조로 접힌다.

```yaml
tags:
  - project/AX-2
  - project/AX-2/module1
  - type/회의록
  - team/생산관리
```

### 권장 태그 카테고리

| 카테고리 | 예시 |
|---------|------|
| 프로젝트 | `project/AX-2`, `project/창신` |
| 문서 유형 | `type/회의록`, `type/요구사항`, `type/MOC`, `type/용어집` |
| 모듈·영역 | `Module1`, `Module2`, `납기예측`, `물류` |
| 팀·역할 | `team/생산관리`, `team/품질` |
| 상태 | `status/draft`, `status/review` (프론트매터 `status` 필드와 중복 안 되게) |

## 6. 체크박스 · 태스크

진행 상황 추적이 의미 있는 항목(구현 기능, 질문 리스트, 액션 아이템)은 체크박스로 쓴다.

### 기본

```markdown
- [ ] 미완료 항목
- [x] 완료된 항목
```

### 옵시디언 확장 상태 (Tasks 플러그인 호환)

아래는 많은 테마·플러그인이 지원하는 커스텀 상태다. 기본 옵시디언에서도 텍스트로는 잘 보인다.

```markdown
- [ ] 할 일
- [/] 진행 중
- [x] 완료
- [-] 취소
- [?] 질문
- [!] 중요
- [*] 별표
```

### 사용 원칙

- 구현 기능 체크리스트, 질문 리스트, 회의 액션 아이템 → 항상 체크박스.

## 8. 문서 구조 전략

### 짧은 단일 노트 (< 300줄)

- 상단: 프론트매터 → 제목 → 개요/요약 콜아웃
- 본문: 섹션별 헤딩 (`##`)
- 하단: 관련 노트 링크

### 긴 문서 (모듈 단위, 섹션이 많은 경우)

**한 파일에 몰지 말고 폴더로 분리**한다:

```
폴더명/
├── 00 - 인덱스 (MOC).md       ← 허브 역할, 다른 문서 링크 모음
├── 01 - 첫 번째 주제.md
├── 02 - 두 번째 주제.md
└── ...
```

- MOC(Map of Content) 문서는 **목차·탐색 허브** 역할.
- 각 개별 문서 상단에 `parent: "[[00 - 인덱스 (MOC)]]"` 프론트매터 추가.
- 문서 끝에 "이전 / 다음" 내비게이션 링크 권장.

### MOC 문서 템플릿

```markdown
---
title: 프로젝트명 (Index)
tags:
  - type/MOC
  - project/...
created: YYYY-MM-DD
---

# 프로젝트명

> [!abstract] 한 줄 요약
> ...

## 문서 구성

1. [[01 - ...]]
2. [[02 - ...]]
...
```

### 네비게이션 링크

문서 하단 패턴:

```markdown
> 상위: [[00 - ...]]
> 이전: [[01 - ...]]   다음: [[03 - ...]]
```

## 9. 파일·폴더 네이밍

### 파일명 규칙

- 공백은 허용 (옵시디언에서 문제 없음). `-` 또는 `·`로 구분자 활용 가능.
- 한글 파일명 OK.
- 같은 폴더 내 순서가 중요하면 **번호 prefix**: `00 - `, `01 - `, `02 - `.
- 날짜가 핵심이면 `YYYY-MM-DD - 제목.md` 형식 (일일 노트, 회의록 등).
- 확장자는 항상 `.md`.

### 폴더 구조

- 주제·프로젝트별로 묶는 것을 권장.
- 첨부 파일(이미지·pdf)은 같은 폴더에 두거나 `attachments/` 하위 폴더에 모아 놓는다.

### 피해야 할 이름

- 양쪽 대괄호 `[...]` — 위키링크 문법과 충돌 가능.
- 맨 앞에 `.` — 숨김 파일 처리될 수 있음.
- 너무 긴 이름 — 위키링크가 장황해진다.

## 10. 이미지·첨부

- 이미지는 가급적 Vault 내부(또는 하위 폴더)에 두고 `![[이미지.png]]`로 임베드.
- 외부 URL 이미지는 `![alt](https://...)` 표준 마크다운 사용.
- 크기 조정: `![[이미지.png|500]]` (가로 500px).
- 원본 파일을 본문 위키링크로 참조해도 된다 (pdf, pptx 등).

## 11. 하지 말 것 · 흔한 실수

- ❌ 본문에 raw HTML 태그 남발 (`<div>`, `<span>`) — 꼭 필요할 때만.
- ❌ 절대 경로 링크 (`/Users/.../file.md`) — **위키링크 사용**.
- ❌ 프론트매터 없이 노트 생성.
- ❌ 프론트매터에 `"[[...]]"`를 따옴표 없이 넣기 — YAML 파싱 깨진다.
- ❌ 이모지 남용 — 꼭 필요한 강조에만.
- ❌ 과도한 굵게/기울임 — 섹션 헤딩으로 푸는 걸 먼저 고려.
- ❌ 같은 내용을 여러 파일에 복제 — 한 곳에 두고 임베드·링크로 참조.

## 12. 작업 흐름 권장안

### 새 문서를 만들 때

1. 비슷한 기존 문서가 있는지 검색.
2. 들어갈 폴더 결정 (없으면 신규 생성 여부 판단).
3. 파일명 결정 (번호 prefix 필요 여부 확인).
4. 프론트매터 작성: 필수 + 이 노트에 의미 있는 선택 필드.
5. 헤딩 구조 먼저 잡고, 콜아웃·표·체크박스를 적절히 배치.
6. 관련 기존 노트에 **역방향 링크** 걸기 (그래프 뷰 연결망 강화).

### 긴 자료(PPT·PDF 등)를 옵시디언화할 때

1. 원본을 `source` 프론트매터에 기록.
2. 한 파일 합본으로 먼저 작성해도 되지만, 모듈·챕터가 3개 이상이면 **폴더 + MOC 구조**로 분리하는 편이 낫다.
3. 분리한 경우 합본은 지우거나 아카이브한다 (중복 관리 비용 ↑).

### 기존 문서를 수정할 때

- 프론트매터에 `updated: YYYY-MM-DD` 필드 갱신.
- 큰 구조 변경 시 기존 위키링크가 끊기지 않는지 확인 (파일명 변경 시 옵시디언이 링크를 자동 갱신해주지만, Claude가 편집할 땐 수동 확인 필요).
