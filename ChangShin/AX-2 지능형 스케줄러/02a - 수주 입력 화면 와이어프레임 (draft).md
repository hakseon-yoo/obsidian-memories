# 수주 입력 화면 1차 와이어프레임 (draft)

> [!abstract] 이 문서의 쓰임
> 2026-04-30 클라이언트 답변(수주 시나리오 = 기제품/신제품 2-way 분기, 입력은 자재코드+수량) 기반의 **1차 화면 골격**. 2026-05-07 15시 "수주 확정 후 프로세스" 미팅에서 STEP 4 → STEP 5(확정 이후) 흐름을 클라이언트와 함께 구체화하기 위한 도구.
>
> **확정 영역 (변경 가능성 낮음)**: STEP 1 시나리오 선택, STEP 2 자재코드+수량 입력

> [!warning] 미확정 항목
> - 신제품 STEP 3 의 금형 메타 입력 항목 (소재/난이도/크기 분류 체계)
> - STEP 4 예상 납기 응답 포맷 (단일 일자? 범위? 신뢰도?)
> - 수주 확정 권한 주체 (영업 / 생산관리 / 자동)
> - 수주번호 채번 규칙

## 1. 시나리오 분기 요약 (출처: 2026-04-30 클라이언트 답변)

| 시나리오 | 진입 조건 | 입력 차이 |
|----------|-----------|-----------|
| **A. 기제품 수주** | 납품 이력이 있는 SKU. 후가공만 변경되는 신제품도 포함 | 자재코드 + 수량 |
| **B. 신제품 수주** | 납품 이력 없음. 금형 설계+개발 일정 추가됨 | "가장 유사한 제품" 자재코드 (참조용) + 수량 + 신제품 메타 |

## 3. STEP 별 상세

### STEP 1 · 시나리오 선택

| 필드 | 타입 | 비고 |
|------|------|------|
| `scenario` | `existing` \| `new` | 라디오 또는 카드형 선택 |

> [!note] UX 가이드
> - 기본값: `existing` (기제품) — 일상 발주 케이스가 다수일 것으로 추정
> - "후가공만 변경되는 신제품도 기제품 수주" 라는 클라 답변을 도움말 텍스트로 노출

---

### STEP 2 · 자재코드 + 수량

#### 공통 필드

| 필드 | 타입 | 검증 | 비고 |
|------|------|------|------|
| `skuCode` | string | required, SKU 마스터 존재 | 자동완성 검색 |
| `quantity` | int | required, > 0 | |
| `requestedDeliveryDate` | date | optional | 클라이언트 명시 안 됨 — 미팅 확인 필요 |

#### 시나리오별 라벨

| 시나리오 | `skuCode` 라벨 | 도움말 |
|----------|---------------|--------|
| 기제품 (A) | "자재코드" | 납품 이력이 있는 제품을 선택하세요 |
| 신제품 (B) | **"유사 제품 자재코드"** | 가장 유사한 기존 제품을 선택하세요. 예상 납기 산출 기준으로 사용됩니다 |

> [!warning] 신제품 라벨이 핵심
> 신제품 흐름에서 "자재코드" 만 노출하면 사용자가 "이 코드 = 만들 제품" 으로 오해할 수 있음. **"유사 제품"** 임을 라벨·도움말·placeholder 모두에 반복 노출.

---

### STEP 3 · 신제품 메타 (B 시나리오 only)

> [!warning] 항목 구조 미정
> 클라이언트 답변에서 "금형 소재, 난이도, 크기, 업체별 capa" 가 변동 변수로 언급됐지만, **각 항목의 enum 값/체계는 아직 미정**. 미팅 또는 후속 인터뷰에서 확정.

| 필드 (가설) | 타입 | 비고 |
|-------------|------|------|
| `moldMaterial` | enum (TBD) | 클라이언트 분류 체계 확인 필요 |
| `moldDifficulty` | enum (low/mid/high?) | TBD |
| `moldSizeCategory` | enum (TBD) | TBD |
| `preferredVendorId` | int? | optional. 외주처 마스터 참조 |

---

### STEP 4 · 예상 납기일 응답

> [!warning] 응답 포맷 미정
> 단일 일자 vs 범위 vs 신뢰도 표기 — 미팅에서 협의.

**1차 응답 영역 후보**

```
┌──────────────────────────────────────────────┐
│ 예상 납기일                                    │
│ 2026-07-15 (47일 소요)                        │
│                                                │
│ 산출 근거:                                      │
│  · 사출 공정: 12일 (외주처 A, capa 70% 점유)     │
│  · 후가공:    8일                                │
│  · 도장:      10일 (외주처 B, capa 90% 점유 ⚠) │
│  · 조립:      7일                                │
│  · 출하 버퍼:  10일                              │
│                                                │
│ [수주 확정]          [취소]                      │
└──────────────────────────────────────────────┘
```

> 산출 근거 노출 여부는 클라이언트와 협의. 사용자가 "왜 이 일자가 나왔는가" 를 이해할 수 있어야 신뢰도 높음.

## 4. 데이터 모델 시사점 (Module 1 entity 후보)

이 화면이 요구하는 데이터/엔티티 (미팅 전 추정):

- `Order` — scenario(enum), customerId, orderedAt, confirmedAt, status, createdBy
- `OrderItem` — orderId, **skuId 또는 referenceSkuId(신제품)**, quantity, requestedDeliveryDate, estimatedDeliveryDate
- `OrderMoldRequest` (신제품 only) — orderItemId, moldMaterial, moldDifficulty, moldSizeCategory, preferredVendorId
- `OrderDeliveryEstimate` — orderItemId, estimatedDate, breakdown(json: 공정별 일자/근거), calculatedAt

> 골격 정리는 Phase 3B 작업 (미팅 후) 으로 분리.
