# 수주 입력 화면 — HTML 1차 산출물 기준 + 보완 항목

> [!abstract] 산출물 정의 (2026-05-07 갱신)
> **1차 산출물(Source of Truth)**: 동료 개발자가 작성한 HTML — `/Users/yoohakseon/Documents/GitLab/changshin/documents/창신-ax2.html`
> 본 문서는 더 이상 와이어프레임 마스터가 아니라 **HTML 에 적용한 보완 항목 + 미해결 검토 사항** 만 추적한다. 화면 마스터는 HTML.
>
> 변경 사유: HTML 이 PPT 워크플로우 6개 워크스페이스(영업·생산관리·생산·품질·구매·자재) 전체를 커버하므로 단일 화면 와이어프레임보다 범위·정합성이 우수. 클라이언트 답변(2026-04-30) 보완 사항만 패치 적용.

## 0. HTML 에 적용된 보완 (2026-05-07)

> 위치: HTML 의 `mNewOrder` 모달 (line 1264~) + 하단 `<script>` 의 `onScenarioChange()` (line 1548~)

| # | 보완 항목 | 적용 방식 |
|---|----------|-----------|
| 1 | 시나리오 enum value 명시 | `<option value="existing/new/reproduction">` 으로 코드 측 키 정렬 |
| 2 | 신제품 선택 시 **SKU 라벨 동적 변경** ("SKU" → "유사 제품 SKU") | `onScenarioChange(value)` JS, `skuLabel` / `skuHint` 갱신 |
| 3 | 신제품 선택 시 **금형 메타 입력 영역 노출** (소재 / 난이도 / 크기 / 선호 외주처) | `#moldMetaSection` 토글. enum 옵션은 가설값 + "분류 체계 미확정" 주석 |
| 4 | 모달 하단 컨텍스트 안내에 **"외주처 생산계획 미수신 → 시스템 내 기수주 점유율 기반 지연 추정"** 문구 추가 | `.ctx` 영역 |

## 2. (참고) 클라이언트 답변 핵심 — 2026-04-30

> 출처: [[Meetings/2026-04-30 - 클라이언트 답변 - 수주 시나리오 · 납기 로직]]

| 시나리오 | 입력 | 흐름 차이 |
|----------|------|-----------|
| **A. 기제품 수주** | 자재코드 + 수량 | 일반 생산 플로우. 후가공만 변경되는 신제품도 이 흐름 |
| **B. 신제품 수주** | "가장 유사한 제품" 자재코드 + 수량 | 금형 설계 + 개발 일정 추가 (평균 45일 · 변동 큼) |

납기 로직은 **앞으로 구현해야 하는 부분** — 변수: 금형 소재 / 난이도 / 크기 / **업체별 capa(1순위)**. 기제품은 부자재·공정별 평균 시간 변수를 클라가 생산관리팀장 인터뷰 후 공유 예정. **외주처 생산계획 직접 수신 불가** → 시스템 내 기수주 점유율 기반 지연 추정.

## 3. STEP 별 상세

### STEP 1 · 시나리오 선택

| 필드 | 타입 | 비고 |
|------|------|------|
| `scenario` | `existing` \| `new` | 라디오 또는 카드형 선택 |

> [!note] UX 가이드
> - 기본값: `existing` (기제품) — 일상 발주 케이스가 다수일 것으로 추정
> - "후가공만 변경되는 신제품도 기제품 수주" 라는 클라 답변을 도움말 텍스트로 노출

---

### STEP 2 · 자재코드 + 수량

#### 공통 필드

| 필드 | 타입 | 검증 | 비고 |
|------|------|------|------|
| `skuCode` | string | required, SKU 마스터 존재 | 자동완성 검색 |
| `quantity` | int | required, > 0 | |
| `requestedDeliveryDate` | date | optional | 클라이언트 명시 안 됨 — 미팅 확인 필요 |

#### 시나리오별 라벨

| 시나리오 | `skuCode` 라벨 | 도움말 |
|----------|---------------|--------|
| 기제품 (A) | "자재코드" | 납품 이력이 있는 제품을 선택하세요 |
| 신제품 (B) | **"유사 제품 자재코드"** | 가장 유사한 기존 제품을 선택하세요. 예상 납기 산출 기준으로 사용됩니다 |

> [!warning] 신제품 라벨이 핵심
> 신제품 흐름에서 "자재코드" 만 노출하면 사용자가 "이 코드 = 만들 제품" 으로 오해할 수 있음. **"유사 제품"** 임을 라벨·도움말·placeholder 모두에 반복 노출.

### STEP 4 · 예상 납기일 응답

> [!warning] 응답 포맷 미정
> 단일 일자 vs 범위 vs 신뢰도 표기 — 미팅에서 협의.

**1차 응답 영역 후보**

```
┌──────────────────────────────────────────────┐
│ 예상 납기일                                    │
│ 2026-07-15 (47일 소요)                        │
│                                                │
│ 산출 근거:                                      │
│  · 사출 공정: 12일 (외주처 A, capa 70% 점유)     │
│  · 후가공:    8일                                │
│  · 도장:      10일 (외주처 B, capa 90% 점유 ⚠) │
│  · 조립:      7일                                │
│  · 출하 버퍼:  10일                              │
│                                                │
│ [수주 확정]          [취소]                      │
└──────────────────────────────────────────────┘
```

> 산출 근거 노출 여부는 클라이언트와 협의. 사용자가 "왜 이 일자가 나왔는가" 를 이해할 수 있어야 신뢰도 높음.

## 4. 데이터 모델 시사점 (Module 1 entity 후보)
